If we use `tokio::task::yield_now().await` inside `execute`, we would have to make `execute` and all its helpers `async`.
This is a huge refactoring and completely breaks WASM which expects synchronous execution (though WASM can support async with `wasm_bindgen(futures)`).
Wait, does it block the async thread?
If `execute` is synchronous, yes it blocks the tokio worker thread. If it blocks the worker thread, the future cannot be dropped because tokio only checks cancellation at `.await` points!
Oh, so Axum CANNOT drop the handler until `g.execute()` returns if it's called directly in the handler without yielding!

Wait, `yagdb`'s issue description: "When a user cancels a request (by closing the connection), stop the query it is running."
Is this reproducible right now?
If I run a very slow query, does it block tokio?
Yes, `Graph::execute` is a CPU-bound operation. If it runs for 10 seconds, the tokio worker thread is blocked for 10 seconds.
If the user closes the connection during these 10 seconds, `hyper`/`axum` won't even process the disconnect for this task until `execute` returns, or it might process the disconnect on another tokio worker thread, but the handler's task won't drop because it's stuck inside the synchronous `execute` method!

To fix this, we can:
1. Make `Graph::execute` check an `AtomicBool` frequently.
2. In `handle_query` and `handle_query_stream`, we set up an `AtomicBool`. We set it to `true` when the request is cancelled.
But wait! If the handler is stuck in `g.execute()`, the handler future will never be dropped because it never hits an `.await` point!
Is this true? Let's write a test to check if a tokio task can be aborted while running CPU-bound sync code if another task calls `abort()`.
No, tokio tasks are cooperatively scheduled. `abort()` only works at `.await` points!
If we can't abort at `.await` points, then the handler task will NOT drop.
BUT, axum might not even drop the future if it's blocked.
Wait, if axum is polling the handler future, it calls `poll`. The future blocks the thread. Tokio will not be able to do anything on that thread.

To allow axum to drop the handler future, we must yield. We can't yield from a sync function.
Wait, we CAN use `tokio::task::spawn_blocking`!
If we use `spawn_blocking`, the tokio worker thread is NOT blocked. It `.await`s the `JoinHandle`.
When the client disconnects, axum drops the handler future. This drops the `.await` on the `JoinHandle`, which detaches the blocking task.
BUT we can also use a `DropGuard` in the handler future to set an `AtomicBool` to `true`! The blocking task reads the `AtomicBool` and stops!

Wait! `graph` is wrapped in `Arc<tokio::sync::Mutex<Graph>>`!
You cannot call `graph.lock().await` and pass the `MutexGuard` across `spawn_blocking`!
`MutexGuard` is NOT `Send` for `std::sync::Mutex`, and `tokio::sync::MutexGuard` IS `Send`, but it's tied to the lifetime of the lock, and passing it to `spawn_blocking` is tricky because it holds an async lock inside a sync thread. Wait, if it's `tokio::sync::MutexGuard<'static, Graph>`, it can be moved to `spawn_blocking`.
Is it `'static`? Yes, `Arc<tokio::sync::Mutex<Graph>>` allows `OwnedMutexGuard`, which can be sent to `spawn_blocking`!
`let mut g = graph.clone().lock_owned().await;`
Yes! `tokio::sync::Mutex::lock_owned()` returns an `OwnedMutexGuard<Graph>`, which is `'static` and `Send`!

Let's test this in `main.rs`:
```rust
async fn handle_query(State(graph): State<SharedGraph>, body: String) -> impl IntoResponse {
    let cancel = Arc::new(std::sync::atomic::AtomicBool::new(false));
    let cancel_clone = cancel.clone();

    struct CancelGuard(Arc<std::sync::atomic::AtomicBool>);
    impl Drop for CancelGuard {
        fn drop(&mut self) {
            self.0.store(true, std::sync::atomic::Ordering::Relaxed);
        }
    }
    let _guard = CancelGuard(cancel);

    let mut g = graph.lock_owned().await;

    // Set the cancel flag on the graph!
    g.cancel_flag = cancel_clone;

    let res = tokio::task::spawn_blocking(move || {
        let result = g.execute(&body);
        // Reset cancel_flag
        g.cancel_flag = Arc::new(std::sync::atomic::AtomicBool::new(false));
        result
    }).await.unwrap();

    match res {
        Ok(result) => (StatusCode::OK, result).into_response(),
        Err(e) => (StatusCode::BAD_REQUEST, format!("Error: {}", e)).into_response(),
    }
}
```
